# Этапы 2-12: План реализации в проекте

## Этап 2. Формы и удобство ввода
Цель:
- Сделать документы и НСИ удобными для ежедневной работы.

Результат этапа:
- Формы списка/объекта приведены к рабочему виду.
- Автоподсчет `Сумма = Количество * Цена` в табличных частях.
- Итог документа считается автоматически.
- Значения по умолчанию для статуса заказа и ответственного.

Проверка:
- Пользователь создает документы без ручной “борьбы” с формой.

## Этап 3. Проведение складских документов
Цель:
- Запустить фактическое движение остатков.

Результат этапа:
- `ПоступлениеТоваров` делает приход в `ОстаткиТоваров`.
- `ПеремещениеТоваров` делает расход/приход по складам.
- `СписаниеТоваров` делает расход.
- Реализован запрет отрицательных остатков.

Проверка:
- Остатки меняются корректно после проведения документов.

## Этап 4. Резервирование по заказу клиента
Цель:
- Реализовать логику резерва товара под заказ.

Результат этапа:
- `ЗаказКлиента` делает движения в `РезервыТоваров`.
- Поддержан частичный резерв.
- Проверка доступного количества к резерву.
- Статус заказа управляется бизнес-логикой.

Проверка:
- Резерв отражается в регистре и не дает “перерезервировать”.

## Этап 5. Реализация и списание резервов
Цель:
- Корректно отгружать товар и снимать резерв.

Результат этапа:
- `РеализацияТоваров` списывает остатки.
- По `ЗаказОснование` снимаются резервы.
- При недостатке остатка документ не проводится.
- Пользователь получает понятные сообщения об ошибках.

Проверка:
- Невозможно провести реализацию “в минус”.

## Этап 6. Общие модули и переиспользование
Цель:
- Убрать дублирование и разделить ответственность.

Результат этапа:
- Общий модуль `УправлениеЗапасами`.
- Общий модуль `УправлениеЦенами`.
- Общий модуль `ОбщегоНазначения`.
- Бизнес-логика вынесена из форм.

Проверка:
- Проверки и расчеты вызываются централизованно.

## Этап 7. Цены номенклатуры
Цель:
- Настроить актуальные цены для продаж.

Результат этапа:
- Регистр `ЦеныНоменклатуры` используется в прикладной логике.
- Цена подставляется в заказ/реализацию по `ВидЦен`.
- Ошибки по отсутствующей цене обрабатываются.

Проверка:
- По выбранному виду цен строки документа заполняются ценами.

## Этап 8. Отчеты на СКД
Цель:
- Показать аналитическую часть решения.

Результат этапа:
- `ОстаткиТоваровПоСкладам`.
- `ПродажиЗаПериод`.
- `ТоварыВРезерве`.
- `ТопТоваров`.

Проверка:
- Отчеты строятся, параметры периода и складов работают.

## Этап 9. Печатные формы
Цель:
- Подготовить документы к реальному документообороту.

Результат этапа:
- Печатная форма `ЗаказКлиента`.
- Печатная форма `РеализацияТоваров`.
- Вывод шапки, строк и итогов.

Проверка:
- Печать доступна из формы документа и отображает корректные данные.

## Этап 10. Интеграция JSON
Цель:
- Показать базовую интеграцию с внешней системой.

Результат этапа:
- Обработка `ЗагрузкаЗаказовИзJSON`.
- Импорт заказа и строк товаров.
- Протокол ошибок загрузки.
- Нормализация входного контракта.

Проверка:
- Заказ создается из JSON-файла без ручного ввода.

## Этап 11. Роли и права
Цель:
- Разграничить доступ по функциям сотрудников.

Результат этапа:
- Роль `Менеджер`.
- Роль `Кладовщик`.
- Роль `Администратор`.
- Проверка сценариев входа под каждой ролью.

Проверка:
- Пользователь видит только доступные ему функции.

## Этап 12. Финализация и демо-сценарий
Цель:
- Подготовить сильную демонстрацию проекта для собеседования.

Результат этапа:
- Наполненные демо-данные.
- Сквозной сценарий “закупка → резерв → реализация → отчеты”.
- Краткая защита архитектурных решений.

Проверка:
- За 10-15 минут показывается полный бизнес-цикл без ошибок.
